## 1. 프로젝트 개요

### 1.1 서비스명: 길이음(GilIEum)

**네이밍 배경**
- "길": 여행 경로, 코스 → 사용자에게 최적의 여행 길을 제공
- "이음": 잇다, 연결하다 → 관광지·맛집·사람을 하나의 코스로 이어주는 서비스
- **핵심 메시지**: "여행의 모든 길을 이어드립니다"

> ※ 기존 "온길"에서 **상표권 이슈**로 "길이음"으로 변경.  
> ※ 내부 팀 분류는 기존대로 **온길(여행팀)** / **온식(맛집팀)**으로 유지.

> ⚠️ **TODO**: 서비스명 "길이음"의 의미가 기획서·발표자료·랜딩페이지 등 모든 곳에 일관되게 녹아들어야 함 (ver02 피드백 반영)

### 1.2 서비스 소개

**길이음**은 사용자의 **연령대, 성별, 동행 유형(가족/친구/커플 등), 인원수, 이동수단** 등 개인 특성을 기반으로 **순천 지역**의 관광지 코스와 맛집을 하나의 여정으로 이어주는 AI 여행 추천 서비스.

### 1.3 핵심 차별점

1. **자체 LLM & RAG 기반 추천** — 범용 검색(네이버/카카오)이 아닌, 자체 DB와 RAG를 결합한 맞춤 추천
2. **지속적 강화 학습** — 사용자 별점·리뷰·체류시간 데이터를 지속 수집하여 AI가 자체 학습, 추천 품질 향상
3. **체류시간 기반 코스 최적화** — GPS 타임스탬프로 각 관광지 체류시간을 측정하여 동일 조건 사용자에게 더 정확한 일정 제안
4. **지역 상생 모델** — 순천 전통시장·소상공인과 연계한 지역 활성화 구조

### 1.4 타겟 지역

- **1차**: 순천시 (프로토타입 ~ 순천시청 협의)
- **2차**: 전남권 확장 (순천시 검증 후)
- **3차**: 전국 확장

## 3. 주요 기능

### 3.1 맛집 추천 (기본 기능)

1. 별점 / 리뷰
2. 영업시간
3. 식당 위치 & 현재 위치 기준 거리
4. 편의시설 및 서비스 정보
5. 블로그 크롤링 기반 "내돈내산" 리뷰 분석

### 3.2 관광 코스 추천

- 사용자 입력 정보 기반 최적 코스 자동 생성
- 코스별 예상 소요 시간 표시
- 코스 내 관광지 → 맛집 → 카페 → 숙소 연계 추천

### 3.3 AI 학습 & 피드백 루프

1. 사용자가 코스·맛집에 **별점** 부여
2. GPS 타임스탬프로 **실제 체류시간** 수집
3. 예측 시간 vs 실제 시간 **차이를 학습 데이터**로 활용
4. 비슷한 조건(연령대·동행유형 등)의 다음 사용자에게 **개선된 추천** 제공

### 3.4 추가 기능 (버전 업그레이드 시)

- 매표소/주요 관광지 사진 촬영 → 카페·숙소 할인권 제공 (앱 활성화 유도)
- 챗봇 UI (기존 Claude/Gemini 히스토리 방식이 아닌, 차별화된 UI)
- 개인화 리뷰·이미지 업로드
- 관리자 설정 페이지 (input/output 관리, 추천 알고리즘 파라미터 조정)

## 4. 기술 아키텍처

### 4.1 기술 스택 (통일안)

> **Django로 통일**: Django는 풍부한 생태계(ORM, Admin, Auth, DRF 등)를 갖추고 있어 빠른 개발과 관리자 기능 구현에 유리. 프론트엔드는 모바일 특성상 React Native(Expo) 유지.

| 계층 | 기술 | 선정 이유 |
|------|------|-----------|
| **프론트엔드 (앱)** | React Native + Expo | 크로스 플랫폼, 빠른 프로토타이핑 |
| **백엔드 API** | Django + Django REST Framework | 풍부한 생태계, Admin 기본 제공, 빠른 개발 |
| **데이터베이스** | MongoDB (주 DB) + Supabase (인증·실시간) | 유연한 스키마(관광데이터), 인증 빌트인 |
| **메시지 큐** | Redis 또는 RabbitMQ | 비동기 작업 처리 (크롤링, AI 추론 큐잉) |
| **컨테이너** | Docker + Docker Compose | 배포 환경 일관성 |
| **클라우드** | AWS / GCP / NCP 중 택 1 | 로컬 서버 지양, 클라우드 우선 (ver02 피드백) |
| **AI/LLM** | 한국형 오픈소스 LLM (아래 상세) | 자체 LLM & RAG 필수 요건 충족 |

> ⚠️ **Kafka**: 현재 규모에서는 과도한 기술. MVP 단계에서는 Redis 또는 RabbitMQ로 충분. 트래픽 증가 시 재검토.

### 4.2 자체 LLM & RAG 전략

> ⚠️ **원본에 구체적 계획이 없어 추가한 섹션**

**모델 후보 (한국형 오픈소스)**
- KoAlpaca, KULLM, Polyglot-Ko 등 한국어 특화 오픈소스 LLM
- 또는 skt/ko-gpt 계열, KAIST 모델 등

**RAG 파이프라인**
```
[사용자 질의] 
    → [Embedding Model] (한국어 임베딩: KR-SBERT 등)
    → [Vector DB 검색] (Chroma / Pinecone / Milvus)
    → [관련 문서 추출] (자체 DB: 크롤링 맛집 데이터 + 공공 관광 데이터)
    → [LLM 프롬프트 생성 & 응답]
    → [코스 추천 결과 반환]
```

**단계별 적용 계획**
| 단계 | 내용 | 시기 |
|------|------|------|
| 1단계 | 외부 LLM API(Claude/GPT) + 자체 DB RAG로 프로토타입 | MVP 기간 |
| 2단계 | 한국형 오픈소스 LLM 파인튜닝 + 클라우드 배포 | 1학기 |
| 3단계 | 사용자 피드백 기반 지속 학습 체계 구축 | 2학기 |

### 4.3 데이터 수집 전략

| 데이터 유형 | 수집 방법 | 우선순위 |
|-------------|-----------|----------|
| 맛집 데이터 | 네이버 블로그 크롤링 (내돈내산 위주) + 네이버 API | 상 |
| 관광지 데이터 | 공공데이터포털 관광 API + 순천시청 공개 데이터 | 상 |
| 영업시간·위치 | 네이버/카카오 API | 상 |
| 사용자 리뷰·별점 | 자체 앱 내 수집 | 중 (서비스 런칭 후) |
| 체류시간 | GPS 타임스탬프 (앱 백그라운드) | 중 (서비스 런칭 후) |
| 숙소 데이터 | 공공데이터 + 제휴 | 하 (2차 버전) |

> **원본 피드백 반영**: "DB를 1순위로 하지 말고, 빠르게 데이터를 수집할 수 있는 API로 모은 다음 데이터 가공·정제를 빠르게 처리할 수 있는 알고리즘을 짜야 함"

## 5. 팀 구조 및 역할

> 서비스명은 **길이음**이지만, 내부 팀 분류는 **온길(여행팀)** / **온식(맛집팀)**으로 운영

### 5.1 온길팀 (여행 — 1팀)

| 항목 | 내용 |
|------|------|

| 항목 | 내용 |

### 5.3 공통 영역

| 영역 | 내용 | 비고 |
|------|------|------|
| **인프라/배포** | Docker, 클라우드, CI/CD | 두 팀 공유 인프라 |
| **Django 백엔드** | 프로젝트 구조, 인증, 공통 모델 | 하나의 Django 프로젝트에서 앱(app)으로 분리 |
| **LLM/RAG** | 자체 LLM & RAG 파이프라인 | 온길·온식 모두 동일 AI 엔진 활용 |
| **사용자 입력 폼** | 연령대, 성별, 동행자 수, 이동수단 등 | 공통 입력 → 각 팀 엔진에 전달 |
| **디자인** | Figma 화면 설계, UI/UX | 디자인과 학생 협업 (가능 시) |
| **DB 설계** | MongoDB + Supabase 공통 스키마 | 관광지·맛집 컬렉션 분리하되 통합 조회 가능하게 |

### 5.4 통합 포인트

- 두 팀의 추천 결과가 하나의 **통합 코스**로 합쳐져야 함 (관광지 → 맛집 → 카페 순서 등)
- **Django 앱 구조**: `ongil` 앱(여행) + `onsik` 앱(맛집) + `common` 앱(공통) 으로 분리
- API 인터페이스 규격을 사전에 합의하여 독립 개발 후 통합
- 아키텍처 다이어그램은 **한 장**으로 표현하되, 좌측=여행/우측=맛집으로 분할 (ver02 피드백)

## 6. MVP 계획

> 로그인/회원가입 없이, 코스 추천 핵심 기능만 빠르게 구현하는 것이 목표

### 6.1 MVP 원칙

- **로그인/회원가입 없음** — 코스 추천만 나오게끔
- 디자인: **Figma로 먼저** 화면 설계 후 구현
- 소요 기간: **1~2개월**
- 두 팀(온길/온식) 각각 MVP 산출물을 만들되, 공통 부분은 함께 진행

### 6.2 온길팀 MVP (여행 코스)

| 기능 | 포함 여부 | 설명 |
|------|-----------|------|
| 사용자 입력 폼 (연령대, 성별, 인원, 이동수단) | ✅ | 공통 입력 폼에서 받은 데이터 활용 |
| 순천 관광 코스 추천 | ✅ | **핵심 기능** — 입력 기반 코스 자동 생성 |
| 지도에 코스 표시 | ✅ | 추천된 관광지를 지도에 마커·경로로 표시 |
| 코스별 예상 소요 시간 | ✅ | 관광지 간 이동시간 + 체류 예상시간 |
| GPS 체류시간 측정 | ❌ | 2차 버전 |
| 경로 최적화 (TSP) | ❌ | 2차 버전 |

### 6.3 온식팀 MVP (맛집 추천)

| 기능 | 포함 여부 | 설명 |
|------|-----------|------|
| 순천 맛집 추천 | ✅ | **핵심 기능** — 코스 내 맛집 연동 추천 |
| 맛집 기본 정보 표시 | ✅ | 별점, 영업시간, 위치, 거리 |
